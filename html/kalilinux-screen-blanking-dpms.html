KaliLinuxことはじめ

Kali Linux の画面がすぐに真っ黒になる（DPMS と電源管理を止める）

質問

Kali Linux にログインしたあと、キーボードもマウスも操作しないでいると数分で画面が真っ黒になる。動画の視聴中や、コマンドの実行結果をぼんやり眺めているときにも勝手に消えてしまう。ターミナルから xset -dpms を打つと、その場では消灯しなくなる（このセッションが続く限りは効く）。そこで「ログインのたびに自動でこのコマンドを実行させたい」と思って、~/.bashrc・~/.profile・~/.zprofile・~/.zshrc・/etc/rc.local など思いつく初期化ファイルに片っ端から書き足してみた。しかし再ログインや再起動の後に確認するとまた消えるようになっている。どこに書けば毎回自動で効くようにできるのか。

原因（理由）

「書き場所がぜんぶ違っていた」のと「xset の設定を後から上書きする別プロセスがいる」の 2 つが同時に起きていることが多い。順に整理する。

  • ~/.bashrc / ~/.zshrc は対話シェル起動時にしか読まれない。GUI でログインしてデスクトップ（Xfce など）が立ち上がる経路では実行されない。だからここに xset -dpms を書いても、「ターミナルを開いた瞬間だけ」効く形になり、GUI セッション全体には反映されない。
  • ~/.profile / ~/.zprofile は「ログインシェル」用。Kali の既定の lightdm（GUI ログイン画面）から Xfce セッションを開始する経路では、シェルではなく X セッションスクリプトが起動するため、これらのファイルは通常読まれない（テキストログインからの startx など、限られた経路でしか実行されない）。
  • /etc/rc.local は X が起動するより前に systemd から実行される。xset は X サーバに対して話しかけるコマンドなので、まだ X が無い段階で実行されると unable to open display エラーで失敗してそのまま終わる。DISPLAY 環境変数も認証情報（~/.Xauthority）もない段階だからどうやっても効かない。
  • そもそも画面消灯の仕組みが複数あり、別々に止めないと黒くなる。
    • X 組み込みの DPMS（Display Power Management Signaling）。xset -dpms で無効化する。
    • X 組み込みのスクリーンセーバ。xset s off（タイマー停止）と xset s noblank（ブランク描画停止）で止める。DPMS と独立しているので両方やる必要がある。
    • Xfce の電源管理デーモン xfce4-power-manager。これが「最大の上書き元」で、自身が独自に DPMS の有効化／タイムアウト値を管理している。ログイン直後にユーザーが xset で切っても、しばらくすると xfce4-power-manager が自分の設定で DPMS を再び有効化してしまう。これを切らない限り xset -dpms を ~/.bashrc 等に書いても泥沼になる。
    • 画面ロッカー light-locker や xscreensaver。これらが入っていると DPMS とは別の経路で画面をブラックアウト＆ロックする。

つまり「正しい場所に書く」だけでなく、「他に消そうとしてくるプロセスをまず黙らせる」ことが必要。以下の対策では、最初に GUI 電源管理を切り（恒久的な上書き元を止め）、次に念のため X セッション開始時に xset を流す形で二重に対策する。

対策（手順）

1. 現状を確認する

まずは xset q で「いま DPMS とスクリーンセーバがどう設定されているか」を見る。ここで読めるようになっておくと、対策後に「ちゃんと無効化できているか」を自分で確認できる。

kali@kali:~$ xset q

主要な出力部分（抜粋）：

Screen Saver:
  prefer blanking:  yes    allow exposures:  yes
  timeout:  600    cycle:  600
...
DPMS (Display Power Management Signaling):
  Standby: 600    Suspend: 600    Off: 600
  DPMS is Enabled
  Monitor is On

読み方：

  • Screen Saver: timeout: 600 = 600 秒（10 分）何も操作しないと X 組み込みスクリーンセーバが発動する
  • prefer blanking: yes = 発動時に画面を真っ黒に塗りつぶす（これが「黒くなる」の正体の一つ）
  • DPMS is Enabled = モニタ電源管理が有効。Standby/Suspend/Off の各秒数で段階的にモニタへ省電力信号を送る

対策後にもう一度 xset q を実行して、DPMS is Disabled／timeout: 0／prefer blanking: no になっていれば成功、というのが確認の指針になる。

2. Xfce の電源管理を GUI で無効化する（推奨）

Kali のデフォルトデスクトップ Xfce では、xfce4-power-manager が DPMS を独自に管理している。これが xset の設定を上書きしてくるので、まずここを止めるのが最優先。

「アプリケーション」メニュー →「設定」→「電源管理」を開く。コマンドからも起動できる：

kali@kali:~$ xfce4-power-manager-settings

開いたウィンドウは上部にタブが並んでいる。次の通り設定する：

  • 「ディスプレイ」タブ
    • 「ディスプレイ電源管理を有効にする」（Display power management）のチェックを 外す
    • その下の「ブランク表示までの時間」「モニタをスリープ状態にするまでの時間」「モニタの電源を切るまでの時間」のスライダはすべて左端（無効 ／ 0 分）に寄せる
  • 「セキュリティ」タブ
    • 「セッションがアイドルになったら画面をロックする」（Automatically lock the session）を 「決して」 に設定する

右下の「閉じる」を押すと設定はその場で保存され、再ログイン後も保持される。

確認のため、もう一度 xset q を実行する：

kali@kali:~$ xset q

出力中の DPMS セクションが DPMS is Disabled に変わっていれば、xfce4-power-manager 側はもう画面を消しに来ない。

3. ログインのたびに xset を自動実行させる（~/.xsessionrc）

手順 2 だけで止まる環境も多いが、別の更新で電源管理が再有効化されたり、X 組み込みスクリーンセーバ（xset s）が独立に動いたりすることがある。保険として、X セッションが開始されるたびに xset を流す 仕組みを入れておく。

Debian 系（Kali を含む）の GUI ログイン経路で確実に読まれるのは、ホームディレクトリの ~/.xsessionrc である。~/.bashrc や ~/.profile ではなく、これを作る。

テキストエディタで新規作成する：

kali@kali:~$ nano ~/.xsessionrc

ファイルの中身として、次の 3 行を入力する：

# X セッション開始時に画面消灯を抑止する
xset -dpms       # DPMS（モニタ電源管理）を無効化
xset s off       # X 組み込みスクリーンセーバのタイマー停止
xset s noblank   # ブランク描画も停止

保存して閉じる（nano の場合）：

  • Ctrl+O を押し、続けて Enter（ファイル名そのままで保存）
  • Ctrl+X でエディタを終了

実行ビット（chmod +x）は不要。X セッション起動スクリプト（/etc/X11/Xsession）が ~/.xsessionrc を source で読み込んでくれる仕組みだから、シェルスクリプトとして実行可能である必要はない。

反映には一度ログアウトして再ログインする：

  • 画面右上のユーザー名／電源アイコン →「ログアウト」→「ログアウト」を選ぶ
  • lightdm のログイン画面が表示されたら、再度ログインする

再ログイン後、ターミナルを開いて確認：

kali@kali:~$ xset q

出力中に次の表記があれば成功：

  • DPMS is Disabled
  • Screen Saver: timeout: 0 かつ prefer blanking: no

4. それでも数分で黒くなる場合：画面ロッカーを確認する

DPMS とスクリーンセーバを両方止めても、別に動いている画面ロッカーが「真っ黒＆ロック」をかけにくることがある。代表的なのは light-locker（lightdm 付属）と xscreensaver。動いているか確認する：

kali@kali:~$ pgrep -l light-locker
1422 light-locker
kali@kali:~$ pgrep -l xscreensaver

番号付きで名前が出れば、そのプロセスが動いている（上の例では light-locker が動いていて、xscreensaver は何も出ないので動いていない）。

light-locker が動いている場合：手順 2 の Xfce「電源管理」設定で「セキュリティ」タブの自動ロックを「決して」にしていれば、light-locker もアイドルロックを発動しなくなる。すでに動いているプロセスを今すぐ止めたい場合は、いったんログアウトして再ログインすれば設定が反映された状態で起動し直る。

xscreensaver が動いている場合：いま動いているものを止める：

kali@kali:~$ xscreensaver-command -exit

次回ログイン時にも起動させたくない場合は、「アプリケーション」メニュー →「設定」→「セッションと起動」→「自動開始アプリケーション」タブを開き、xscreensaver の行のチェックを外す。

注意点

  • なぜ .bashrc / .profile / /etc/rc.local に書いても効かないのか、3 行まとめ：.bashrc は GUI ログイン経路では実行されない／.profile は lightdm から Xfce を起動する経路では読まれない／/etc/rc.local は X 起動前に走るので xset がそもそも繋がる先が無い。GUI セッション開始時に確実に読まれる場所は ~/.xsessionrc。
  • 設定を入れたあとは xset q で必ず再確認する。DPMS is Disabled と timeout: 0 が揃っていれば、少なくとも X とユーザー設定の側からは画面を消しに行かない状態になっている。
  • GNOME / KDE では電源管理デーモンが違う。Kali を Xfce 以外の派生版で使っている場合、上書きしてくるのは xfce4-power-manager ではなく gnome-settings-daemon（GNOME）／powerdevil（KDE）になる。GUI 設定の場所は各デスクトップ環境の「電源管理」「省電力」「ディスプレイ」設定にあるので、そこで「画面の電源を切る」「ブランクスクリーン」を 無効 ／しない に変更する。~/.xsessionrc の方は同じファイルでそのまま使える。
  • VirtualBox 上で動かしている場合、ホスト OS 側の電源設定も影響する。Kali（ゲスト）の DPMS を完全に止めても、Windows（ホスト）の「設定」→「電源とバッテリー」→「画面とスリープ」で「画面をオフにする」が短時間に設定されていると、ホストの画面が消えてゲストも見えなくなる。両方の設定を見ること。
  • Wayland セッションでは xset 自体が効かない。Kali Xfce のデフォルトは X11 なので通常は関係ないが、念のため確認したい場合はターミナルで echo $XDG_SESSION_TYPE を実行し、出力が x11 なら xset 系の手順がそのまま効く。wayland と出る場合は手順 3 が無効で、各デスクトップ環境の GUI 電源管理設定（手順 2）のみで止める必要がある。
  • 学習用 VM 向けの設定であり、業務環境ではそのままコピーしないこと。画面の自動ブランク／自動ロックは席を離れた間の覗き見対策として重要な機能。共用 PC や業務利用では「画面ロック（手順 2 のセキュリティタブ）は残し、ディスプレイ電源管理（同・ディスプレイタブ）だけ無効化する」運用も検討する。
